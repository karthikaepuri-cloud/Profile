Migration · Architecture
ECC vs S/4HANA
Complete Comparison
By Karthik Apuri, SAP Enterprise Architect · Brisbane, Australia · 2025

1. Core Overview

Full NameSAP ERP Central ComponentSAP Business Suite 4 SAP HANA
Released2004February 2015
DatabaseAny RDBMS (Oracle, MSSQL, DB2, MaxDB)SAP HANA only — mandatory
ProcessingBatch-heavy; aggregates pre-computed overnightReal-time; in-memory column store; no pre-aggregation

Extended maintenance is available until 2030 at additional cost. Over 20,000 SAP customers still need to migrate. Planning and execution typically take 12–36 months — do not underestimate.

2. Architecture Differences

ECC uses a traditional 3-tier architecture with row-based database storage. All analytical workloads require a separate BW system. S/4HANA uses SAP HANA's in-memory column store to serve both transactional and analytical workloads simultaneously (HTAP).

🏛️ ECC Stack

SAPGUI → AS ABAP → ABAP Dictionary (row tables) → Any RDBMS + Separate BW for Analytics

⚡ S/4HANA Stack

Fiori (HTML5) → AS ABAP + CDS Views → HANA push-down → SAP HANA (column store) + Embedded Analytics

LayerECCS/4HANA
UISAPGUI, Web DynproSAP Fiori, Fiori Launchpad
DB LayerOpen SQL, minimal push-downABAP CDS + HANA push-down (logic in DB)
Table StorageRow-based transparent tables + many aggregate tablesColumn-based; no pre-aggregated totals
APIsRFC / BAPI (function modules)OData v4 / REST; SAP API Business Hub
IntegrationSAP PI/PO, ALE/IDOCSAP Integration Suite (BTP), Event Mesh
ExtensibilityABAP mods, user exits, BADIsRAP, BTP side-by-side, Key User Tools

3. Data Model Simplification

The most profound change in S/4HANA is the Universal Journal (ACDOCA). In ECC, a single financial posting updated 20+ tables. In S/4HANA it writes once to ACDOCA, and HANA computes all aggregations on demand — reducing database size by 50–80%.

DomainECCS/4HANA
FI JournalBKPF + BSEG + 20+ aggregates (GLT0, KNC1, LFC1…)Single table: ACDOCA
CO ControllingCOEP, COSP, COSS, COBK — separate CO tablesMerged into ACDOCA — FI+CO unified
Customer/VendorKNA1 (customer) + LFA1 (vendor) — separateBusiness Partner (BUT000) — unified
Profit CenterGLT0 (separate EC-PCA ledger)Eliminated — all in ACDOCA
Asset AccountingANLA/ANLB/ANLC; batch depreciationReal-time via ACDOCA extension ledger
CO-PACosting-based OR account-based (duplicated)Account-based only; no duplication

5. Migration Paths

🔄 Brownfield (System Conversion)

Convert ECC in-place. Data and config preserved. Custom code must be adapted per Simplification List. Lowest disruption, fastest path, but carries forward technical debt.

🌱 Greenfield (New Implementation)

Fresh S/4HANA using SAP Best Practices. Only open items and master data migrated. Maximum redesign opportunity. Highest effort, best long-term result.

🔀 Bluefield (Selective Data Transition)

Hybrid — convert some processes brownfield, re-implement others greenfield. Tools: SNP Transformation Backbone, SAP Landscape Transformation. Most flexible, most complex to govern.

6. Migration Prerequisites

These are the prerequisites that must be met before migration can begin, organised by migration path. Items marked Critical must be resolved before any migration activity starts.

🔄 Brownfield Prerequisites

SYSTEM CONVERSION
Critical
SAP ECC 6.0 EHP 6 Minimum
Must be on ECC 6.0 EHP 6 or higher. EHP 7/8 strongly preferred. Earlier versions require a separate EHP upgrade project before migration can begin.
Critical
Maintenance Planner Validation
SAP Maintenance Planner must produce a valid stack XML without errors. All add-ons must have S/4HANA-compatible versions or be decommissioned.
Critical
FI-CO Reconciliation = Zero
All FI-CO reconciliation differences must be resolved to zero. Run FINS_MIG_RECON_CHECK. Even a single cent difference will block the migration tool (SUM).
Critical
Business Partner Migration
All Customers (KNA1) and Vendors (LFA1) must be converted to the Business Partner model (BUT000). Run BUPA_PRE_MIGRATION. This is mandatory — S/4 cannot function without it.
Critical
Custom Code Remediation (ATC)
Run ABAP Test Cockpit with S/4HANA check variant. All Priority 1 findings (obsolete APIs, SELECT * on BSEG, aggregate table access) must be resolved before go-live.
Critical
SAP HANA Infrastructure Provisioned
Target HANA system provisioned, sized, and certified (TDI or appliance). HANA version must match S/4HANA target release. No other database is accepted.
Important
Simplification List Review (600+ Items)
Review SAP Note 2270399. Identify which Simplification Items impact active business processes and plan remediation. Scope this as its own workstream.
Important
New Asset Accounting Preparation
Run FINS_ACDOC_UPGRADE checks. All asset periods must be closed. Parallel valuation areas must be restructured per New AA design before migration.
Important
CO-PA Migration to Account-Based
Costing-based CO-PA is deprecated in S/4. If active, migrate to account-based CO-PA. Involves redesigning characteristics and value fields into G/L account logic.
Important
Material Ledger Activation
Material Ledger is mandatory in S/4HANA. If not active in ECC, it must be activated before migration. Can have P&L implications — test thoroughly in sandbox first.
Standard
Data Archiving (ILM)
Archive historical data to reduce database size and migration runtime. SAP ILM recommended. Can reduce technical migration window significantly for systems over 5TB.
Standard
Interface Inventory & Validation
Document all inbound/outbound interfaces (IDOCs, RFCs, BAPIs, file). Verify target systems can connect post-migration. Some BAPIs are replaced by OData APIs in S/4.
Standard
Authorisation Role Review
New S/4 transactions and Fiori apps require new authorisation objects. Review all PFCG roles. Business Role concept replaces many technical roles. Re-run SoD analysis.

🌱 Greenfield Prerequisites

NEW IMPLEMENTATION
Critical
Business Process Blueprint Signed Off
Complete process redesign aligned to SAP Best Practices. Fit-Gap workshop outputs must be approved before any configuration begins. No approved blueprint = no build.
Critical
Master Data Cleansing & BP Conversion
All master data (customers, vendors, materials, CoA, cost centres) must be cleansed, de-duplicated, and mapped to S/4HANA data model. Business Partner conversion mandatory.
Critical
Chart of Accounts Design Finalised
New CoA structure must be finalised before open item migration. G/L account mapping from ECC to S/4 documented. Account-based CO-PA characteristic design must align to new CoA.
Critical
Open Item Migration Strategy Defined
Cutover strategy for open AR, AP, and GL items using LTMC (Legacy Transfer Migration Cockpit). All open items must balance on migration day. Dry run mandatory.
Critical
Full DEV/QAS/PRD Landscape on HANA
Entire 3-system landscape (or more) must be provisioned and verified on SAP HANA. Cloud or on-premise — all systems. No legacy DB systems in the landscape.
Important
Interface Redesign to OData/REST
All interfaces should be redesigned to OData/REST using SAP Integration Suite where possible. PI/PO iFlows must be rearchitected — do not lift and shift.
Important
Authorisation Redesign (from scratch)
Design roles from scratch using S/4HANA Business Role concept. Do not copy ECC roles. Fiori apps require different object classes. Update SoD ruleset for S/4 objects.
Standard
Historical Data Archiving Decision
Decide what historical data (if any) to migrate. Most greenfields migrate 1–2 years. Archive rest in read-only ECC (SAP ILM). Document legal retention requirements per country.
Standard
Change Management & Training Plan
Greenfield = new system for users. Comprehensive change management and role-based training is mandatory. Start communications 6+ months before go-live.

🔀 Bluefield Prerequisites

SELECTIVE DATA TRANSITION
Critical
Entity / Scope Partitioning Defined
Clearly define which company codes / entities / modules will be converted (brownfield) vs re-implemented (greenfield). This scope must be locked before tooling selection.
Critical
Transformation Tool Selected & Contracted
Select tool early: SNP Transformation Backbone, Rimini Bluefield, or SAP Landscape Transformation (SLT). Each has different licensing, timelines, and data handling approaches.
Critical
All Brownfield Prerequisites (for converted entities)
For every entity or module being converted, all Brownfield prerequisites (FI-CO recon, BP migration, ATC, HANA infra) still apply in full.
Critical
All Greenfield Prerequisites (for new entities)
For every entity or module being re-implemented, all Greenfield prerequisites (blueprint, master data, CoA, LTMC) apply in full for those entities.
Important
Data Isolation & Carve-out Plan
Define how data from converted entities is isolated from re-implemented entities. Cross-entity dependencies (intercompany, shared master data) must be explicitly mapped.
Important
Cutover Complexity Planning
Bluefield cutover is significantly more complex than either pure path. Requires a detailed cutover runbook with entity-by-entity sequencing, rollback points, and reconciliation steps.
Standard
Governance Model for Hybrid Period
During migration, some entities operate in ECC while others are in S/4. Establish a hybrid governance model — change freeze policies, dual-landscape monitoring, and user access control.

7. What the Readiness Check Misses

💡 About This Section

SAP's standard Readiness Check (via SAP for Me / Maintenance Planner) and the Custom Code Migration app cover the technical code and add-on compatibility. But there are significant areas that require manual discovery — organisational, configurational, and integration risks that no automated tool will flag. This is where experienced architects add the most value.

🔧 Custom Configuration & Enhancement Framework 6 checks
▼
⚙️
Implicit Enhancements in Standard Programs
Implicit enhancements embedded directly into standard SAP code (not classic user exits) are invisible to the Custom Code Migration app. They exist in the source system but have no object entry in the repository that a standard scan can catch.
How to find: Use SE80 → Search for "Enhancement Implementations" in Repository Browser. Also run: SUIM → Programs by Authorization Object and manually review SMOD/CMOD entries. Query table ENHLOG and cross-reference with TADIR for Z* implementations on standard includes.
📋
Substitution & Validation Rules (GGB0 / GGB1)
Financial substitutions and validations configured in GGB0/GGB1 often contain hardcoded account numbers, company codes, or document types that won't exist after migration or CoA changes. The Readiness Check does not evaluate these rules.
How to find: T-code GGB0 (Substitutions) and GGB1 (Validations). Export all rules and review field references against your S/4HANA data model and new CoA mapping. Pay special attention to BSEG field references — that table is replaced by ACDOCA.
🖨️
SAPscript / SmartForms / Adobe Forms Output
Legacy output forms (SAPscript, SmartForms) are not flagged by ATC as broken, but many reference standard programs, function modules, or structures that are retired in S/4HANA. They will compile but fail at runtime.
How to find: Run SE71 (SAPscript) and SSF_FB_DOC_UPLOAD (SmartForms) inventory. Query NAST and TNAPR for active output types. Map each form to its driver program and check that program against the Simplification List. Test all forms in sandbox post-migration before UAT.
🔢
Number Range Objects
Number range objects for FI documents, material docs, and SD orders may conflict post-migration, especially if the Universal Journal requires a single unified number range that didn't exist in ECC. This causes silent data corruption risks.
How to find: T-code SNRO — export all number range objects and current intervals. Cross-check against SAP Note 2349012 (Number ranges for S/4HANA Finance). Specifically check BELNR (FI document number) — S/4 changes number range grouping logic.
📑
Custom Posting Keys & Document Types
Custom posting keys and document types created in ECC may not map cleanly to S/4HANA's revised document classification logic. In particular, asset-related document types behave differently under New Asset Accounting.
How to find: T-codes OB41 (Posting Keys) and OBA7 (Document Types). Export all Z* / customer-specific entries and validate each against the New AA documentation and SAP Note 2220005 (New Asset Accounting configuration checklist).
📊
BAdI Implementations on Deprecated Function Modules
Classic BAdI implementations (SE18/SE19) that call deprecated function modules or access BSEG, BSID, BSIK, GLT0, or other aggregate tables will pass ATC but fail at runtime in S/4HANA because those tables are empty or removed.
How to find: T-code SE18 — export all active BAdI implementations. For each, open the implementing class in SE24 and search for calls to: GET_BSEG, READ TABLE BSEG, GLT0, BSID, BSIK, KNC1, LFC1. Cross-reference with ACDOCA field mapping in SAP Note 2182260.
🔌 Interfaces & Integration 5 checks
▼
🤝
RFC Destinations Calling ECC-Only Function Modules
External systems calling SAP via RFC (legacy MES, WMS, payroll systems) often call function modules that are renamed, deprecated, or replaced by BAPIs/OData in S/4HANA. The calling system won't know until it calls and gets BAPIRET2 errors or wrong data.
How to find: T-code SM59 — export all active RFC destinations. Then use SE37 → Where-Used List for each called FM and check against the Simplification List. Also query table ARFCSSTATE and TRFCQIN for recently called FMs from external systems to build an inventory of what's actually in use.
📨
IDOC Message Types with Retired Segments
IDOC-based interfaces using message types like INVOIC, ORDERS, DESADV may reference segment types that are changed or removed in S/4HANA. The PI/PO mapping may produce IDOCs that technically post but populate incorrect fields in the Universal Journal.
How to find: T-code WE20 / WE21 — export all active partner profiles and message types. Run BD53 to check reduction message types. Cross-reference each IDOC segment against the S/4HANA IDOC compatibility matrix in SAP Note 2267010. Flag any segment using fields sourced from BSEG or VBKD that no longer exist.
📁
File-Based Interfaces with Hardcoded SAP Paths
Legacy file-based interfaces (outbound ALV extracts, FTP drops, payroll file outputs) often hardcode SAP server OS paths. After migration the system hostname, SID, and mount paths change — silently breaking all file-based integration.
How to find: Search all Z-programs and scheduled jobs for hardcoded paths using SE80 Full Text Search with patterns like '/usr/sap/', '/sapmnt/', or the current SID. Also check SM37 for active background jobs writing to external paths, and review FILE (logical file names) in T-code FILE — these often aren't updated during migrations.
🔑
Stored Credentials in SM59 / Destinations
RFC and HTTP destinations in SM59 often contain hardcoded usernames and passwords for service accounts. These accounts may be system users that are decommissioned post-migration, or the password policy changes — causing silent authentication failures at go-live.
How to find: T-code SM59 — export all destinations with "Basic Authentication" type. Cross-reference user IDs with SU01 to verify they are active service accounts. Coordinate with security team to rotate and vault credentials using a secrets manager or SNC (Secure Network Communications) before go-live.
🌐
PI/PO Mappings Referencing Deprecated Structures
SAP PI/PO message mappings that reference BSEG, BKPF, VBKD or other ECC-specific structures will continue to technically "work" post-migration but return empty fields or incorrect data — a dangerous, hard-to-detect failure mode.
How to find: In PI/PO (ESR), export all active Integration Objects. For each message mapping, open the source/target structures and flag any reference to: BSEG, BSID, BSIK, BSAD, GLT0, KNC1, LFC1, VBKD segment fields. Map these to the equivalent ACDOCA / VBAP fields and update mappings in sandbox before migration.
👥 Organisational & Master Data 4 checks
▼
🗂️
Duplicate Business Partners Pre-Migration
The Business Partner migration tool will create one BP per KNA1/LFA1 entry. If the same real-world company exists as both a customer AND a vendor in ECC (common in intercompany scenarios), migration creates two separate BPs — breaking reconciliation and dunning workflows.
How to find: Run BUPA_PRE_MIGRATION in check mode first — it identifies duplicates. Also write a custom ABAP report comparing KNA1.NAME1+KNA1.ORT01 against LFA1.NAME1+LFA1.ORT01 with fuzzy matching. Resolve all duplicates using FSCM_BUPA_MERGE before executing the actual BP migration.
💱
Open Items in Obsolete Currencies or Exchange Rate Types
Open AR/AP/GL items using exchange rate types or currency codes that are deprecated, renamed, or remapped in S/4HANA will fail FI-CO reconciliation checks and block migration. This is especially common in multi-currency / multi-ledger landscapes.
How to find: Query BSID and BSIK for all open items and extract WAERS (currency) and KURSF (exchange rate). Cross-reference against table TCURC and TCURR for validity. Run FAGL_CHECK_TRANS and review the output for currency inconsistencies before executing any migration run.
📦
Material Master with Incomplete Views or Old Industry Sectors
Material masters with missing mandatory S/4HANA views (e.g. MRP 3, Plant Data / Storage 1) or using industry sectors that are restructured in S/4 will cause MM transactions to fail post-migration. The Readiness Check does not validate material master completeness.
How to find: Run MM60 (Material Usage) to find materials with zero movements — candidate for archiving. Use SE16 on MARA to query MTART (material type) and MBRSH (industry sector) for non-standard values. Cross-check against SAP Note 2399104 (Material Master fields changed in S/4). Mass maintain missing views using MM17 in sandbox.
🏢
Cost Centre / Profit Centre Hierarchy Gaps
S/4HANA's Universal Journal requires every posting to carry both a Cost Centre and Profit Centre. In ECC it was possible to post to accounts without Profit Centre assignment. These gaps become hard errors in S/4 and will block document posting if not resolved pre-migration.
How to find: Run report FAGLB03 (G/L Account Balances) filtered for Profit Centre = blank. Also query BSEG WHERE PRCTR = '' AND BELNR NOT LIKE 'INT%'. For Cost Centres, query CSKS for entries without valid hierarchy assignment. Resolve all gaps using OKB9 (default account assignment) and OKKS (profit centre settings) before migration.
🔐 Security & Authorisation 3 checks
▼
🛡️
Roles Referencing Deleted / Renamed Transactions
Hundreds of ECC T-codes are deleted, renamed, or replaced by Fiori apps in S/4HANA. Roles containing these T-codes will not throw an error in PFCG but will silently grant access to non-existent transactions, or fail to grant access to the S/4HANA equivalents.
How to find: Export all roles via SUIM → Roles by Transaction. Compare T-codes against SAP's Fiori Apps Reference Library (SAP Note 2870261) and the Deleted T-code list in the Simplification Item for your target release. Use the Fiori Tracker tool or the Access Control module in SAP Cloud ALM to map ECC roles to Fiori app equivalents.
🔒
Background Job Service Users with Dialog Authorisations
Background jobs in ECC often run under dialog user IDs with broad A_AUTH or S_DEVELOP access. S/4HANA's security hardening rejects dialog-type authorisations in batch context for several new auth objects, causing overnight jobs to fail silently.
How to find: T-code SM37 — export all scheduled jobs and extract the executing user IDs. For each, run SU01D to check user type (Dialog vs System). Flag all Dialog users running batch. Review their role assignments for S_DEVELOP, A_AUTH, and any new S/4-specific auth objects from SAP Note 2308217. Convert to System user type where appropriate.
🌐
ICF Services Enabled Beyond Standard Requirements
ICF (Internet Communication Framework) services that were activated in ECC for legacy portals, BSPs, or third-party connectors may represent security vulnerabilities in S/4HANA, or may be deprecated and replaced by Fiori / OData paths. The Readiness Check does not audit ICF service activation.
How to find: T-code SICF — export all active (green) ICF services. Cross-reference against the S/4HANA ICF service list in SAP Note 1485029. Flag all Z* services and legacy BSP paths. Deactivate any service not explicitly needed — follow SAP Security Guide for S/4HANA (available on SAP Help). This is especially critical for internet-facing systems.
📊 Reporting & Analytics 3 checks
▼
📈
Report Painter / Writer Reports on Retired Tables
Report Painter and Report Writer reports built on ledger groups that reference GLT0, COSS, COSP, or EC-PCA tables will produce zero results or errors in S/4HANA because those tables are no longer populated. These reports are invisible to ATC.
How to find: T-code GRR3 (Report Painter) and FGI3 (Report Writer) — export all custom reports. Check the library each report uses: libraries 0F, 8A2 (GLT0-based) are incompatible with S/4. Must be rebuilt using Fiori analytical apps, CDS-based reports, or SAP Analytics Cloud live models on ACDOCA.
🔢
ALV Reports with Hardcoded BSEG / Aggregate Table SELECTs
Custom ALV reports (Z-programs) doing direct SELECT from BSEG, BSIS, BSAS, GLT0 etc. will either return empty results (tables exist but are empty in S/4) or throw dumps. ATC may flag some of these, but reports using dynamic SQL or field catalog tricks often escape detection.
How to find: Use SE80 Full Text Search across all Z-programs for: "FROM BSEG", "FROM GLT0", "FROM BSID", "FROM BSIK", "FROM KNC1". Also search for OPEN SQL joins to these tables. Map all findings to the equivalent ACDOCA CDS views. Prioritise by usage frequency — pull from SM20 / STAD for actual usage data.
🔗
BW Extractors Pointing to Deprecated DataSources
If BW is kept alongside S/4HANA for historical reporting, BW extractors that use ECC-era DataSources (0FI_GL_4, 0FI_AP_4 etc.) must be replaced by S/4HANA-specific extractors. Running old extractors against S/4 returns incorrect or no data without any error message.
How to find: In BW, use RSA1 → DataSources → filter on source system = your ECC SID. Export all active DataSources. Cross-reference against SAP Note 2333488 (BW DataSources for S/4HANA) to identify which must be replaced. S/4HANA-specific DataSources follow naming pattern 0FI_GL_14, 0FI_AP_16 etc. Plan BW migration/upgrade as part of the S/4HANA project scope.
✅ Key Takeaway — What experienced architects do differently

The automated Readiness Check gives you ~40% of the picture. The remaining 60% — substitution rules, output forms, IDOC segment compatibility, role gaps, Report Painter libraries, and BW extractor mappings — requires a structured manual assessment. Building a complete inventory across these 5 categories before the migration project kicks off is what separates a smooth go-live from a crisis at cutover weekend.
